# Estimate validation and comparison design

## Goal

`/estimate:new` の見積もり入力に含まれる単位・粒度・前提差を可視化し、同一案件の再見積もりで大きな乖離が出た場合に、根拠を比較してからレポートを確定できるようにする。

## Non-goals

- O/M/P の値を自動的に補正・丸め・拒否しない。
- 実績のない状態で、どちらの見積もりが正しいかを計算器が判定しない。
- 外部の承認サービスや永続ストレージを追加しない。

## Design

### 1. 粒度・単位の診断

計算器は O/M/P を時間として受け取る。`o <= m <= p` の既存検証に加え、M が 4 時間未満または 24 時間超のタスクを `granularity_warnings` として返す。

- 4 時間未満: 小タスクである理由、または近接タスクと統合しない理由をレポートで説明する。
- 24 時間超: 分割できない理由をレポートで説明する。
- 診断は警告であり、正当な小タスク・大タスクを失敗させない。
- レポートは O/M/P の単位が時間であり、人日は集計結果だけであることを明記する。

### 2. 実行コンテキストの履歴化

履歴スキーマ v3 を追加する。v2 は引き続き読める。v3 の各タスクレコードには、同一 run で同一の `run_context` と `run_summary` を格納する。別ファイルの run summary は復活させない。

`run_context` は次を持つ。

- `comparison_key`: 同一スコープを比較するための明示キー
- `qa_included`, `ai_view`, `analysis_mode`
- `hours_per_day`, `correlation`
- `sources`: 入力文書の安定した識別子
- `scope`, `exclusions`, `dependencies`, `assumptions`: 比較に必要な短い文字列配列

`run_context` は計算器だけが永続化する。v2 レコードには存在しないため、比較時にはメタデータ不足として扱う。参照クラス補正と実績記録は v2/v3 の双方を対象にする。

`run_summary` は計算器がシミュレーション後に生成し、従来型の P50/P80（時間・人日）、試行回数、分布、seed を持つ。入力前提である `run_context` と分離し、再見積もりのP50差を、レポート解析や再シミュレーションなしで比較できるようにする。

### 3. Run 比較

`estimate_calc.py compare-runs` を追加する。比較キーが一致する二つの run を指定し、以下を返す。

- P50/P80、O/M/P合計、期待値合計の差分
- タスク数、カテゴリ別件数、M 値の分布、粒度警告件数の差分
- `run_context` の項目別差分
- 比較可能性 (`comparable`, `reason`) と、v2・キー不一致などの理由

タスク名の意味的な同一性は機械判定しない。WBS対応とスコープの解釈は、比較結果を読む人がレビューする。

### 4. 再見積もりの照合ゲート

`/estimate:new` は比較対象 run を明示指定できる。候補の `comparison_key` が一致し、比較結果で P50 の差が 30%以上、または run context に差分がある場合、レポート確定前に次を確認する。

1. スコープ・除外・依存関係
2. QA、解析モード、AI表示、換算時間、相関
3. WBS の対応と粒度警告の根拠

差分を解消できない場合、単一の置換値として扱わない。楽観・標準・悲観のシナリオと採用理由をレポートへ記録する。

### 5. レポート表示

既存の固定構造を維持し、以下を追加する。

- 見積もり手法: O/M/Pの時間単位、粒度診断結果
- 前提条件: run context の比較可能な前提
- キャリブレーションの後: 比較対象・差分・照合結果（比較時のみ）
- シナリオ表（照合未解消時のみ）

## Error handling

- `run_context` の型・必須キーが不正な v3 payload は、履歴を書き込む前に失敗する。
- `compare-runs` は run が存在しない、複数の context が混在する、比較キーが異なる、または v2 である理由を構造化して返す。
- 警告は計算・履歴追記を妨げない。

## Tests

- M<4h と M>24h の警告、境界値、警告があっても履歴が書き込まれること
- v2 と v3 の読み取り・参照クラス補正・実績記録の互換性
- v3 context の検証失敗時に履歴が書き込まれないこと
- 同一キーの比較、キー不一致、v2不足、context差分、30%差分
- スキル契約に、単位・context・照合ゲート・未解消時のシナリオ記録が存在すること

## Versioning

計算器・スキル・テストの変更と同じ変更単位で、プラグインバージョンを更新し、CHANGELOG に記載する。
