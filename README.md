# estimate — Claude Code 用の工数見積もりプラグイン

仕様書・Issue・短い作業説明から、根拠のある工数見積もりを作成するプラグインです。

1. 作業を WBS の末端タスクへ分解する
2. 各タスクに楽観・最頻・悲観（3点）の範囲を設定する
3. モンテカルロシミュレーションで集計する
4. 参照クラス予測で、チーム固有の履歴に照らして補正する

結果は一点見積もりではなく、**P50（計画用）** と **P80（コミット用）** です。

利用するほど精度が上がります。作業完了後に実績時間を記録すると、同じカテゴリの次回見積もりのアンカーとバイアス補正に使われます。開始時に履歴は不要ですが、似た作業の実績が蓄積されるほど精度は向上します。

## インストール

```text
/plugin marketplace add chun-mura/estimate-workload
/plugin install estimate@estimate-workload
```

### 更新

```text
/plugin marketplace update estimate-workload
/reload-plugins
```

`.claude-plugin/plugin.json` の `version` を更新すると、新しいバージョンが取得されます。

### ローカル開発

マーケットプレイスを経由せず、プラグインディレクトリを直接マウントします。

```bash
claude --plugin-dir .
```

## 使い方

### 1. 見積もりを作成する

```text
/estimate:new --mode quality docs/spec.md docs/tasks.md
/estimate:new --mode economy docs/spec.md docs/tasks.md
/estimate:new --mode quality --ai-view docs/spec.md docs/tasks.md
/estimate:new --mode economy --no-ai-view docs/spec.md docs/tasks.md
```

1回の実行は、関連するタスク群1件分です。共有分析を繰り返さないため、関連タスクは1回にまとめてください。独立した複数コマンドは、別タスク群として扱われます。

流れは次のとおりです。

1. 入力を読み込み、モードに応じた分析を実行する
2. WBS を作成し、3点見積もりを設定・補正する
3. シミュレーションを実行する
4. `docs/estimates/` 配下へレポートを保存し、`run_id` を表示する

レポートはすべて `skills/new/references/report-template.md` の固定構造に従うため、実行ごとに比較できます。

#### モード（`--mode`）

分析の深さを `--mode quality` または `--mode economy` で指定します。他のオプションと併用でき、順不同です。

| オプション | 内容 |
| --- | --- |
| `--mode quality` | 要件分析とリポジトリ分析を、タスク群全体に対して一度だけ実行する |
| `--mode economy` | 要件分析のみ。コード影響は未検証として扱う |

指定しない場合は、実行時に対話で確認します（未回答なら停止し、既定値は選びません）。

#### AI支援表示（`--ai-view` / `--no-ai-view`）

レポートに [AI支援工数](#2つの工数ビュー) を含めるかどうかを、引数で明示できます。`--mode` と併用でき、順不同です。

| オプション | 内容 |
| --- | --- |
| `--ai-view` | AI支援工数の表示を含める。対話確認をスキップする |
| `--no-ai-view` | AI支援工数の表示を含めない（従来型工数のみ）。対話確認をスキップする |

どちらも指定しない場合は、実行時に対話で確認します（未回答なら含める）。`--ai-view` と `--no-ai-view` は同時に使えません。

#### レポート名

入力ファイル名が `A-AR-001_` のような設計 ID で始まる場合、レポート名は次の形式になります。

```text
YYYY-MM-DD-a-ar-001-<work-name>.md
```

ファイル名から ID を取得できない場合や上書きしたい場合は、`--id a-ar-001` を指定してください。

### 2. 実績を記録する

作業完了後、その実行に実績時間を記録します。

```text
/estimate:record <run_id>
```

実績を履歴へ書き戻し、キャリブレーションの要約（見積もりと実績のバイアス、カテゴリごとの比率）を表示します。

## データとプライバシー

プラグインはプロジェクトローカルの `.estimate/` 配下にデータを保存します。ほかの場所へデータは送信されません。

| パス | 内容 |
| --- | --- |
| `.estimate/history.jsonl` | 唯一の永続見積もり成果物。見積もり済み／完了済みタスクを1行ずつ保存 |
| `.estimate/config.json` | 設定（例: `hours_per_day`） |

`history.jsonl` への書き込みは `append-history` と `update-actual` からのみです。コマンドはレポート用にシミュレーションメタデータを返しますが、これは別の見積もり成果物としては保存されません。

### リポジトリ運用

| 方針 | やること |
| --- | --- |
| チームでキャリブレーションを共有する | `.estimate/history.jsonl` をコミットする |
| 見積もりデータをローカルに留める | `.estimate/history.jsonl` を無視する |

共有履歴がある場合にのみ、メンバーは共有アンカーと補正の恩恵を受けられます。

実行 ID にはタイムスタンプとスラッグが含まれ、ローカル履歴ファイル内でのみ重複排除されます。同じスラッグの作業を同一秒に別クローンで見積もると、後の Git マージで実行 ID が重複する可能性があります。まれですが、共有履歴をマージする際に確認してください。

## 2つの工数ビュー

レポートは既定で、時間と人日それぞれについて P50 / P80 の合計を示します。1人日は 8 時間（`.estimate/config.json` の `hours_per_day` で変更可）です。

| ビュー | 意味 |
| --- | --- |
| **従来型工数** | AI支援による調整なし。見積もりどおりにシミュレーションした合計 |
| **AI支援工数** | カテゴリ別係数でタスクを調整してからシミュレーションした合計 |

AI支援係数は、各カテゴリで AI支援あり・なしの完了実績がそれぞれ 3 件以上になると履歴から求めます。それまでは組み込み既定値を使います。レポートには各係数の出所を記載します。

| 指標 | 意味 | 用途 |
| --- | --- | --- |
| **P50** | 中央値。超過する可能性としない可能性が同程度 | 社内計画 |
| **P80** | 実際の工数が 80% の確率で下回る、より保守的な値 | コミット・対外見積もり |

どちらも「最頻」値の合計ではなく、O/M/P 範囲全体を対象としたモンテカルロシミュレーションの結果です。

## v1 の制約

- **固定カテゴリ体系** — タスクは少数の組み込みカテゴリに分類されます（`skills/new/references/category-taxonomy.md`）。現時点では独自カテゴリを追加できません。
- **フックと MCP は未対応** — スキル、エージェント主体のワークフロー、独立した計算スクリプトで構成されます。Claude Code のフックや MCP サーバーは登録しません。

## バージョニング

`scripts/`・`skills/`・`agents/` のいずれかを変更したら、同じコミットまたは PR で次を更新してください。

- `.claude-plugin/plugin.json` の `version`
- `CHANGELOG.md`

バージョンを据え置いたまま内容だけを変えても、バージョン比較するプラグインマネージャーには未変更と区別できません。キャッシュ済みインストールが古いコードのまま動き続けることがあります。

実際に、初期スキャフォールドから複数の挙動変更を経ても `version` が `0.1.0` のままだったため、キャッシュ済み 0.1.0 が、上流で修正済みの点見積もり式を使い続けた事例があります。

### セマンティックバージョニングと履歴スキーマ

- プラグインはセマンティックバージョニング（`.claude-plugin/plugin.json`）に従います。
- 履歴レコードには `schema_version` があります。リーダーはファイルを書き換えず、未知の `schema_version` は警告付きでスキップします。
- プラグイン 0.1.x が書き込んだ `schema_version: 1` の `pert` フィールドは、シミュレーションモデルと一致しない式で計算されています。現行リーダーは互換性のない基準を混在させないため、アンカーとキャリブレーションから除外します。
